# prepare-exec-summary\n\n## Purpose\n\nUse this workflow to compress a complex topic into an executive-ready summary for a boss, leadership, or another decision-maker.\n\nThis is not a generic summary. It is an upward-facing decision brief focused on:\n- the conclusion\n- why it matters\n- key evidence\n- major risk\n- what support or decision is needed\n\n## Trigger phrases\n\nPrefer this workflow when the user says things like:\n- Help me prepare a summary for my boss.\n- Compress this into one page.\n- How should I say this to leadership?\n- Help me turn this into an executive summary.\n- Make this sound like an upward report.\n- Help me prepare what I should say in the meeting.\n\n## Routing rules\n\nChoose this workflow when one or more of the following is true:\n1. The audience is a boss, leadership, or decision-maker.\n2. The user wants to compress analysis into upward communication.\n3. The user needs \"how to say it,\" not more analysis.\n4. The summary needs clear decisions, risks, or asks.\n\nDo **not** use this workflow as the primary one when:\n- the user still needs the actual analysis -> first run the relevant analysis workflow\n- the user only needs meeting notes -> use a summary / notes path instead\n- the user needs a PRD or product solution document -> use a requirements workflow\n\n## Minimum input\n\nTry to gather:\n- summary topic\n- current conclusion or judgment\n- most important evidence\n- current status or progress\n- main risk or blocker\n- desired decision / ask / support\n- audience\n\nAt minimum, start once you know:\n- the topic\n- the current judgment\n- the audience\n\n## Follow-up policy\n\n### Default number of follow-ups\n- Standard mode: 2-4\n- Fast mode: 1-2\n\n### Highest-priority follow-ups\n1. What do you want the audience to do after hearing this?\n2. What does this audience care about most: outcome, risk, resource, or timing?\n3. What is the single most important conclusion?\n4. What is the strongest evidence behind it?\n5. What is the biggest current risk?\n\n### When not to ask much more\nIf the user already has sufficient analysis but needs sharper upward expression, shift quickly into restructuring. Do not turn the task back into a long diagnosis.\n\n### Critical-premise rule\nIf the summary would materially change based on 1-2 missing facts, ask those first unless time is obviously tight.\nTypical examples:\n- the real purpose of the communication\n- the audience's main concern\n- whether the user needs a decision ask, a risk alert, or a status summary\n\n### When to produce a fast version immediately\nDo it when:\n- the meeting is imminent\n- the user explicitly wants a first version they can say out loud\n- the conclusion and ask are already mostly clear\n\n## Processing logic\n\nFollow this sequence:\n1. Clarify the purpose of this communication.\n2. Distill 1-3 top conclusions.\n3. Decide what needs to be decided now versus validated next.\n4. Keep only the minimum evidence needed to support those conclusions.\n5. Highlight the main risk or blocker.\n6. Clarify the ask, decision point, or support needed.\n7. Produce both a written summary and a spoken version when useful.\n\n## Readiness / launch-call rule\n\nWhen the real issue is launch timing, rollout confidence, or readiness under pressure:\n- structure the answer as a decision call, not a risk memo\n- make clear:\n  - the current recommendation\n  - the readiness gap\n  - the consequence of pushing ahead anyway\n  - what confidence is based on\n  - what support or decision is needed now\n\nAvoid fake confidence, but do not retreat into generic caution language.\n\n## Common failure modes\n\nWeak output usually looks like this:\n- writing polished status notes instead of a decision-ready summary\n- leading with background or process before the conclusion\n- sounding balanced but leaving the actual recommendation fuzzy\n- mentioning risks without saying what should happen because of them\n- implying an ask instead of stating it explicitly\n- including too much detail and too little decision signal\n\n## What strong output looks like\n\nStrong output should:\n- lead with the bottom line in the first screen\n- tie the recommendation to business consequence, timing, or resource logic\n- make the main risk legible without turning the note into a risk register\n- state the ask in approve / challenge / modify terms\n- make decide-now versus validate-next explicit if uncertainty remains\n- read like something leadership can respond to immediately, not just admire\n\n## Output structure\n\nUse this structure when helpful:\n\n1. Summary topic\n2. Conclusion first\n3. Why this matters now\n4. Current situation / key evidence\n5. Main risks or issues\n6. Recommendation and ask\n7. Decide now vs validate next\n8. Direct spoken version\n\n## Output length control\n\n### Fast meeting version\nUse when the user needs something immediately:\n- 1-3 conclusions\n- 1-2 risks\n- 1-2 asks\n- one spoken paragraph\n\n### Standard\nUse for chat, memo, or message drafts:\n- full output structure above\n\n### One-page version\nUse for a more formal exec brief:\n- standard structure plus tighter status and timeline detail\n\n## Success criteria\n\nA good result should:\n- make the conclusion visible immediately\n- avoid reading like a meeting transcript\n- surface meaningful risk\n- include a clear ask or decision point\n- be usable without a lot of extra explanation\n- make it obvious why now, why not later, or why not the alternative\n\n## Failure cases\n\nTreat these as failures:\n1. writing meeting notes instead of an executive summary\n2. leading with process instead of conclusion\n3. listing risks without a recommendation\n4. giving recommendations without enough support\n5. failing to state the ask clearly\n6. drowning the summary in detail\n\n## Notes\n\nExecutives usually do not need the full journey. Prioritize what matters for judgment, resourcing, timing, and risk.\n


## Launch-call type rule

When the real issue is launch timing, rollout confidence, or readiness under pressure, force the recommendation into one of these explicit call types unless the user clearly asked for a looser draft:
- **commit to launch**
- **commit to limited / gated launch**
- **delay launch**
- **do not lock the date yet; lock a decision checkpoint instead**

The output should make clear:
- the current recommendation
- the main readiness gap
- the consequence of pushing ahead anyway
- what support or decision is needed now
- why the other launch-call types are not the current recommendation

## Extra success criterion for launch-pressure communication
- in launch-pressure situations, land on an explicit launch-call type rather than vague "cautious progress" language
